# 🚀 【核心设计】平台后端架构与技术设计文档

## A. 总体技术判断

### 1. 架构路线对比：可配置中台 (Headless CMS) vs 纯自定义后端
针对“产品/技能中心 + 社区体系 + 可后台配置”的需求，业界有两种典型解法：

| 对比维度 | 方案一：Headless CMS（推荐 Directus / Strapi） | 方案二：自定义后端应用（推荐 NestJS + Prisma） |
| :--- | :--- | :--- |
| **核心优势** | 开箱即用，自带功能完善的后台管理UI，自动生成 API，能极快速实现内容配置化诉求。 | 逻辑完全可控，架构极度灵活，最适合处理复杂的“社区发帖、强关联逻辑”等业务场景。 |
| **开发成本** | 前期开发极简，基本通过鼠标配置即可跑通产品和技能模块。 | 前期中等（需写 CRUD 和配合独立的后台前端）。 |
| **社区功能适配度** | 较弱。复杂的点赞、楼中楼评论、防刷机制、状态机流转等，需要重度定制开发，体验不佳。 | 极高。完全贴合具体业务定制，后续无论是接入 AI 审核还是特定的风控拦截都水到渠成。 |
| **长期维护难度** | 基本为“增删改查”时极低，一旦业务产生深度定制，维护核心生命流的成本极高。 | 整体架构稳定清晰，扩展性强，人员好招聘，长期接纳新功能毫无瓶颈限制。 |

### 2. 最终架构推荐
**明确推荐方案：自定义后端路线（NestJS 后端服务 + PostgreSQL + React 或 Vue 现成中后台前端模板）**
*   **推荐理由**：虽然项目需要“后台无代码级更改前端配置”，但“社区系统（帖子审核、管理员下场、举报）”极具业务特殊性。纯粹的 CMS 无法优雅且安全地处理这类复杂状态机。我们通过在 NestJS 中使用 PostgreSQL 的 **JSONB 字段**来解决页面动态配置问题（实现配置下发中心），既获得了 CMS 的低成本灵活性，又保全了代码层面应对未来任何变态级社区交互的扩展力。

## B. 后端需求文档 (PRD)

### 1. 项目目标
将现有的前端原型及展示型系统，升级为由“前端互动层 + API接口网关层 + 持久化数据层 + 管理中后台页面”构成的一体化平台。支持前端模块化的运营热更新，及健全的社区用户互动引擎。

### 2. 系统角色定义 (全平台权限控制基石)
*   **游客角色 (Guest)**：基本浏览公开页面、产品库、公告。
*   **普通用户 (User)**：登录态下可发帖、评论互动、点赞、举报违规内容。
*   **版主/审核员 (Moderator)**：可访问后台部分菜单，负责帖子/评论的审批、退回、置顶、人工下架。
*   **运营人员 (Operator)**：可访问后台内容管理菜单，配置产品和技能的上新/排序、更新“前端页面组件配置”、发官方公告。
*   **超级管理员 (Super Admin)**：系统最高权限，负责后台角色配置、全局日志审计。

### 3. 功能核心模块细化
*   **页面配置中心模块 (Page Engine)**：后端提供键值 JSON 管理接口，供前端动态拉取不同页面的骨架数据（比如首页精选推荐、导航栏、横幅）。实现管理员只要填表，前端立刻易容的热更新目标。
*   **产品/工具库生态模块**：产品的全周期CRUD操作。不仅记录基础信息（名称/图标/外链），支持多对多的关系绑定（如一个产品可下挂多项技能）。后台支持上下架、热度干预。
*   **社区枢纽模块**：
    *   **常规互动**：基于富文本或 Markdown 格式记录用户发布流、楼层式多级评论表、点赞关系表。
    *   **审核干涉机制**：支持将所有新帖设定为 pending 状态进入后台人工待审队列；支持标记`is_official`将该帖升级为高亮官方讨论。
    *   **举报系统**：举报信息归集至后台监控面板以便执行人员阻断。
*   **全局与用户模块**：标准化账户认证；角色树生成机制。

### 4. 异常、日志、安全等运维架构要求
*   **异常拦截器**：全局封装错误逻辑，不管代码报什么错，返给前端的一定是 `{"code": 错误码, "msg": "提示文本", "data": null}`。切断数据库原始错误裸奔外网的风险。
*   **安全要求**：登录态采用去中心化 JWT（携带角色标识）签发；社区发言接口（写库接口）启用防重攻击及访问频次控制 (Rate Limiting) 防刷；用户输入过滤 XSS 富文本校验。
*   **审计动作要求**：所有非 GET 查询的后台核心变更操作，均拦截入库 `admin_actions` 表单。日后谁误删了推荐位，都立档备查。

## C. 数据库设计基石

**选用数据库层：PostgreSQL (`pgsql2`)**
> _战略分析：兼具深度的关系型限制（保障社区和用户模块一致性无坏数据），和优秀的非关系数据（JSONB 类型，完美契合复杂配置需求）。MySQL 虽然普及，但 JSON 处理不尽如人意。_

### 1. 账号模型体系
*   **`users`**：`id` (主键/UUID), `username`, `email`, `password_hash`, `role_id`, `status` (主动枚举: 正常, 封号), `created_at`
*   **`roles`**：`id`, `name` (如 admin/user), `description`
*   **`permissions` & `role_permissions`**：交叉表实现精细控制。

### 2. 动态内容驱动层（前端无代码配置核心）
*   **`page_configs`**：
    *   `id`
    *   `page_key` (唯一标识，如 "home_banner", "sidebar_tools")
    *   `config_data` (**JSONB**：极其重要的核心字段。前端解析这段 JSON 渲染样式，例如 `["name": "工具A","url": "xxx"]`，这直接代替了 CMS 的建表之苦)
    *   `version` 及 `updated_auth_id`

### 3. 业务体现实体模型
*   **`products`**：`id`, `name`, `desc`, `cover_url`, `url`, `status` (已发布/待发布), `sort_weight`, `created_at`
*   **`skills`**：`id`, `title`, `content_md`, `category`, `status`, `created_at`
*   **`product_skill_rel`** (联合绑定关系表)：`product_id`, `skill_id` (联结以上两类业务)。

### 4. 社区管控系统模型
*   **`posts`**：`id`, `user_id`, `title`, `content`, `status` (pending:待审/approved:已发/rejected:打回), `is_pinned` (置顶), `is_official` (官贴), `created_at`, `deleted_at` (必须是软删除，保留数据以便溯源！)
*   **`comments`**：`id`, `post_id`, `parent_id` (支持楼中楼嵌套), `user_id`, `content`, `status`
*   **`tags` & `post_tags`**：社区标签打标。
*   **`reports`**：`id`, `target_type` (枚举：帖/评论/人), `target_id`, `reporter_id`, `reason_text`, `status`
*   **`admin_actions` (审计)**：`id`, `admin_id`, `module`, `action` (动作文本), `target_id`, `ip_address`, `created_at`

## D. API 结构设计理念

API 以 RestFul v1 版本形式透出。并按“展示端（公域）”、“会员端（私享）”与“管理后台端（权域）”区分命名空间与守卫拦截。

| 分类区块 | 核心接口示例 | 请求与职能 | 准入 |
| :--- | :--- | :--- | :--- |
| **动态排版加载** | `GET /api/v1/public/configs/:page_key` | 拉取 JSONB 供前端渲染。入参：页面标识，出参：对应的结构化数据 | 匿名/开放 |
| **基础与验证** | `POST /api/v1/auth/login` | 处理账密认证或未来第三方 SSO。入参：表单，出参：JWT (带权限树) | 开放 |
| **业务承载** | `GET /api/v1/public/products` | 带分页拉取产品清单。核心入参：`page`, `sort=weight` | 开放 |
| **社区创建体系** | `POST /api/v1/users/posts` | 发帖功能。创建后通过库层将其初始 status 强制为 pending | 需身份验证 |
| **后台维护-业务** | `PUT /api/v1/admin/configs/:key` | 管理员更新 JSON 配置的接口。实现“点键即生效”功能 | 高级运营人员 |
| **后台维护-防务** | `POST /api/v1/admin/posts/:id/audit` | 审核批复。入参：批复动作 (approve/reject)，以及理由 | 审核管理员 |

## E. 后台管理系统设计 (The Back-Office)

后台不仅是 CRUD，而是平台的总控中心。建议分离为独立的 React+AntDesign 前端项目对接后方：

1.  **看板仪表盘 (Dashboard)**：显示平台关键指数（今日新增报告产品、当前待审帖子积压量等）。
2.  **前端内容热配置中心**：系统核心！以表单向导甚至低代码方式呈现对应不同 `page_key` 的后台修改器。运营修改上传更新，对应上述 `/api/v1/admin/configs` 更新。
3.  **产品技能引擎页**：包含“上新表单”、“业务详情编排管理”、“关联组装管理（指定哪个产品匹配哪个技能）”。
4.  **云治理社区中心**：
    *   **审核工控台**：双页签“待审列队”与“驳回列箱”。审核员进行决议。
    *   **官场操作台**：支持针对既有内容的直接加权处理（打精华、锁库、加置顶标签）。对举报的高危内容集中批量下架。
5.  **安全设置与系统**：日志展示视图，权限树拖拽分配节点视图。

## F. 数据库接入体系与交付运维

对于现有的三组环境资源（mysql2 / pgsql2 / mongo2），明确：**全盘启用 PostgreSQL 实例。**

1.  **建立网络防卫壁垒**：若后端项目部署在 Hostuno **同区生态**，极度建议透过内网网关直连，切忌裸奔外网，规避抓包勒索。若后端如托管在 Vercel 等无服务或第三方平台运行，需要通过 PostgreSQL 的 SSL 强加密链接机制，并在云防火墙将 Hostuno 相关白名单配置正确。
2.  **环境变量核心范例**：
    ```bash
    # 使用标准连库语句。无需给代码过度配置底层文件。
    DATABASE_URL="postgresql://[用户]:[你的密码]@pgsql2.hostuno.com:5432/[库名]?schema=public"
    ```
3.  **数据生命期规划 (Migrations)**：这是团队协作中最容易失控的地方！代码层必须引用 Prisma Migrate 或者 TypeORM 类似体系。数据库初始化、每一张增加字段的表、每一段索引修改，全部以 `.sql` 追踪入提交历史（Git）。严禁使用类似 Navicat 手搓数据库结构瞒报上线。定期通过 Cron Job 进行 `pg_dump` 归档冷备份防止误彻底物理删除导致灾难。

## G. 实施阶段破局路线图 (Roadmaps)

化整为零迭代开发策略：
*   **第 1 阶段（基建与鉴权）**：跑通后端脚手架，建立 Postgres 链接。编写全局响应、异常过滤器。敲定 RBAC 实现多端的 JWT 生命周期流转。（产出可测试的 Auth 协议接口群）
*   **第 2 阶段（CMS配置与核心引擎）**：重点实操 `page_configs` 的动态下发中心、产品库/技能库体系逻辑 CRUD。（产出后结合现有前端联调，此时前端部分已经彻底动态化）
*   **第 3 阶段（社区中枢建设）**：建立帖子树，盖楼评论系统。落实带有 “打回、审批”的审核流程状态机制。（产出全流程的社区体验测试接口）
*   **第 4 阶段（后台总控UI对接）**：使用中后台模板将以上 API 进行视图化展现。制作仪表盘。运营/测试同学接手进行综合演练实操。
*   **第 5 阶段（生产加固）**：对接口引入频次访问拦截（Redis限流）、日志补强分析、生产级的进程守卫和压测。

## H. 架构定调最后总结

- **推荐组合方案：后向 PostgreSQL → 使用 Node体系 (NestJS 极佳) 面向对象开发 API → UI基于现成中台体系定制。**
- **选型哲学**：用关系型的严谨应对人和权限管理的刚需，用 JSONB 的容差对应多变展示。
- **避免的设计误区**：千万不要因为觉得“配置灵活”就走向完全的无代码后端（如过度沉迷低代码平台），一旦社区功能（反压制、评论折叠算法甚至举报机制升级）上线，复杂的表关系与业务逻辑会成为无代码平台的噩梦。
- **刚性正确准则**：在创建 DB 层时，涉及内容的实体表（帖子、评论、技能介绍）要求一律带 `deleted_at` 等时间戳进行 **【逻辑软删除】**。严控所有对数据库的真实 DELETE 执行权限。
